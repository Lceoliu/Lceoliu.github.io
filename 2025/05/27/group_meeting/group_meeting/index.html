组会报告汇总

写在前面

这篇文档从 Notion 中的组会记录整理而来。原始记录是按日期堆叠的实验汇报，图片、loss 曲线、可视化视频和临时想法很多；这里不逐张搬运图片，而是把研究逻辑、关键实验、阶段性结论和后续方向整理成一条可以回看的主线。

目前这条线索可以概括为：

  1. 从手语识别的 CTC/VAP 对齐问题入手；
  2. 发现仅靠改 loss 难以解决核心问题，瓶颈更可能在 feature 表示、关键点预处理和 decode/force alignment；
  3. 用 pure CTC、人工构造 feature、中心化/归一化可视化、silhouette score 等方法逐步定位问题；
  4. 引入 body part embedding、RoPE、不同部位尺度、时序卷积、ST-GCN/Two Stream 等结构和基线；
  5. 后续研究从 gloss 级识别扩展到 RVQ/Sign Token，再进一步面向长视频和文档级评价。

总体研究问题

项目的输入主要是手语视频中提取出的关键点序列，目标是在弱对齐或无帧级标注的情况下，完成从连续动作到 gloss 序列、再到文本或语言模型输入 token 的建模。

这个问题有几个天然困难：

  • 手语动作和 gloss 之间没有严格的一帧一词对应关系，需要从序列级标注中学习时间对齐。
  • CTC 可以处理无帧级标注的序列识别，但它容易产生 peaky prediction，也就是概率峰集中在少数帧上。
  • 关键点数据的归一化、中心化、尺度选择会直接改变动作信息的可分性。
  • 如果输入 feature 本身不可分，后面的 CTC、beam search、force alignment 很难补救。
  • 当任务从短句识别扩展到长视频或文档级翻译时，传统 WER/BLEU 只能覆盖局部错误，无法充分评价跨句顺序、段落结构和篇章一致性。

因此，研究逻辑不是简单地“堆模型”，而是围绕三个问题反复排查：

  1. 对齐是否可靠：CTC、beam search、force alignment 是否能给出可信的 gloss 时间边界？
  2. feature 是否可分：同一个 gloss 的帧级表示是否真的聚在一起，不同 gloss 是否被拉开？
  3. token 是否可用：如果要把手语动作喂给 LM，VAE/RVQ/Sign Token 的序列化方式是否保留了动作结构？

阶段一：熟悉 pipeline，尝试 VAP 与 CTC

2025.03.01 - 2025.03.07

最开始的工作是熟悉手语项目开发流程，重点阅读 CTC、Non-Peaky CTC 和 VAP（Visual Alignment Pretrain）。当时的目标是先理解现有模型如何从关键点序列得到 gloss 概率，再看能不能通过额外的视觉-文本对齐预训练改善 CTC 的尖峰问题。

CTC 的核心优势是允许输入帧数 \(T\) 和输出 gloss 长度 \(N\) 不一致。模型每一帧输出一个 gloss 分布，CTC 通过 blank 和 collapse 规则把帧级序列压缩成目标 gloss 序列，并对所有可能路径求和。它适合没有帧级边界的数据，但副作用是模型可能只在少数帧上给出极高置信度，其余帧都给 blank，导致可解释的时间边界很差。

VAP 的直觉是让视觉片段和文本/gloss embedding 在预训练阶段先对齐，再进入下游识别。这个方向理论上能缓解 CTC 的弱对齐问题，但实际落地会受到词表、分词和 embedding target 的约束。

2025.03.14

这一周尝试了两个方向：

  • 用 label priors 修正 CTC，即对输出概率额外除以先验概率，试图削弱高频 gloss 的影响；
  • 实现 VAP loss，但发现现有 pipeline 需要较大修改，例如整句词 embedding、QFormer 去留、gloss 词表和外部语言模型分词不一致等。

阶段性结论是：简单 label priors 对 loss 作用不明显；VAP 的主要工程障碍在于 embedding 空间和 gloss 标注体系不完全对齐。比如 mBART 的 tokenizer 会把一个 gloss 切成多个 subword，而我们实际 CTC 训练用的是 gloss 词表，两边长度和语义单位都不一致。后来考虑过用 Sentence-BERT 先对 gloss 词表编码，但这仍然不能直接解决帧级视觉对齐问题。

2025.03.21

VAP 训练跑通后，实际切片效果不理想。这说明“加入一个对齐 loss”并不会自动带来可用的时间边界。此时研究重点开始从“是否能复现 VAP”转向“CTC 的输出到底如何对齐，decode 和 feature 到底哪里出问题”。

阶段二：算法 review，明确瓶颈位置

模型结构

当时的模型可以抽象为：

graph TD
  A["手语关键点 B x T x keypoints x 2"] --> B["MLP feature extractor"]
  B --> C["帧级特征 B x T x D"]
  P["位置嵌入"] --> D["特征 + 位置"]
  C --> D
  D --> Q["QFormer / Transformer Encoder"]
  Q --> H["分类器"]
  H --> O["gloss 概率 B x T x num_gloss"]
  Q --> L["辅助 embedding / SignCL loss"]

关键疑问是：新的数据集中没有稳定的 embedding target，那么 QFormer 的 embedding output 是否还有足够监督？如果 embedding loss 在某些 checkpoint 之后实际上被忽略，那么 QFormer 的这部分设计可能没有发挥预期作用。于是后续实验开始尝试更纯粹的 CTC 结构。

CTC decode 与 force alignment

算法 review 里还整理了 decode 和 force alignment：

  • beam search decode：从每一帧的 gloss 概率矩阵中搜索最可能的 gloss 序列。当前参数大致是 nbest = 1、beam size = 15，后续也考虑通过增大 beam size 找回漏掉的 peak。
  • trellis force alignment：给定概率矩阵和 decode 出来的 gloss 序列，用动态规划回溯每个 gloss 的起止帧。

force alignment 的状态可以理解为 \(k[t,j]\)：在时间 \(t\) 到达第 \(j\) 个 token 的最大概率。递推时比较“前进到下一个 gloss”和“停留在 blank”的两种路径：

\[ k[t+1,j+1] = \max\{k[t,j]p(t+1,G_{j+1}), k[t,j+1]p(t+1,G_{blank})\} \]

样本分析中出现过类似问题：ground truth 是“好、上次、补、牙、后、感觉、有点、酸、疼”，预测变成“好、补、牙、后、觉得、有点、疼”。这说明模型并非完全没学到动作，但容易漏词、混淆近义/近形词，且 force alignment 有时更像把 peaky frame 视为 gloss 的开始，而实际可能更接近动作结束。

阶段性判断：

  • CTC 确实能驱动模型学到部分重要特征；
  • 但 decode 和 force alignment 不是纯后处理问题，它们强依赖概率矩阵质量；
  • 如果帧级 feature 本身不可分，beam search 和 force alignment 只能在坏分布上做近似。

阶段三：Pure CTC 与 feature 瓶颈

2025.06.30

为了排除 QFormer 等复杂模块的影响，开始做 Pure CTC Test。结构简化为：

graph TD
  A["关键点序列"] --> B["MLP feature extractor"]
  B --> C["位置编码 / 相对位置"]
  C --> D["Transformer Encoder"]
  D --> E["MLP classifier"]
  E --> F["CTC gloss probability"]

实验记录里有几个关键结果：

  • Transformer 从 3 层增加到 6 层后，val loss 反而从约 17 变到 18，说明盲目加深模型没有解决瓶颈。
  • 不过滤 gloss 时 loss 约 37，filtered 后约 17，说明词表规模、样本分布和长尾 gloss 对训练稳定性影响很大。
  • 后续方向包括相对位置编码、GCN、可视化 transformer output、缩小 gloss dict 到高频词、用 frame embedding 和 gloss embedding 对齐等。

这里开始明确一个方向：先证明 feature 空间是否能把 gloss 分开，再谈更复杂的识别头。

2025.07.10

这一阶段加入了：

  • body part embedding；
  • RoPE 作为相对位置编码；
  • 对 blank 类设置较低权重的 cross entropy；
  • 训练集过拟合实验；
  • feature 相似度分析。

结果发现，所有 gloss 的特征相似度分布很接近，不是“同一 gloss 内部更接近、不同 gloss 更远”，而是整体都混在一起。这直接指向 feature extractor 或关键点预处理：模型可能没有得到足够可分的输入表示。

2025.07.13

进一步怀疑 mean/std 归一化让 2D 点过于接近，于是尝试：

  • 预处理时不对 2D 点除以 std；
  • 对不同部位的特征做不同尺度放大；
  • 加入简单的时间维卷积；
  • 后续考虑 ST-GCN stream、预训练 backbone、对 transformer output 施加相似度/对比学习 loss、相邻帧变化 loss、多特征融合等。

这一阶段最重要的认识是：对于关键点方法，输入的尺度和中心化不是无害的 preprocessing detail，而是会直接决定动作差异是否还在。

阶段四：归一化、中心化与可分性量化

2025.07.18 - 2025.07.22

围绕关键点归一化做了大量可视化。主要比较了：

  • 每帧固定中心点；
  • 时间平均中心点；
  • 不同 cover ratio：100%、80%、50%；
  • 同一 cover ratio 下不同中心化方法；
  • 同一中心化方法下不同 cover ratio。

当时还用 Mean Silhouette Score 对 feature 可分性做量化。记录中的结果如下：

中心化方法 cover ratio Mean Silhouette Score
固定中心点 100% 0.3775
固定中心点 80% 0.3702
固定中心点 50% 0.3767
时间平均中心点 100% 0.3727
时间平均中心点 80% 0.3765
时间平均中心点 50% 0.3708

这些数值差异不大，说明简单换中心化或 cover ratio 并没有显著提高 gloss 聚类质量。换句话说，问题可能不只是坐标整体平移/缩放，而是需要更强的结构先验，例如关节拓扑、局部部位建模或 RGB/pose 融合。

2025.07.25

这一页是整个排查链路的关键。为了验证“CTC 是否真的受 feature 质量支配”，人为构造了一个“良好 feature”：

  • 用 one-hot 编码每个 gloss 的 feature；
  • 其他维度用随机数填充；
  • 只用一层线性层做分类。

结果显示，当初始 feature 足够可分时，简单分类器也能让 CTC decode 得到合理序列。这说明 CTC 本身不是主要瓶颈。真正的问题是当前 feature extractor 没有把关键点序列映射到一个足够可分的表示空间。

这个结论把后续方向从“继续调 CTC loss”推向“先改善 feature 表示”。

阶段五：外部基线、数据增强与工程问题

2025.07.30

在 Phoenix2014 上和 Two Stream 纯 Keypoint 方法做了对比：

方法 WER
Ours, beam size = 30 49.90774%
Two Stream 纯 Keypoint 30.97%

这说明当前 pure/keypoint pipeline 和成熟 Two Stream baseline 还有明显差距。差距可能来自多个方面：

  • backbone 对局部动作和时序模式的建模不足；
  • 关键点预处理损失了可辨别动作细节；
  • decode 和 force alignment 对 peaky CTC 输出仍然脆弱；
  • 仅 keypoint 信息可能不足，需要 RGB 或更强 pose backbone。

2025.08.04

阅读了一篇关于 pose-only 数据增强 ablation 的论文，并对自己的模型做了类似 ablation。记录中比较了“添加全身关键点”和“无全身关键点”，得到的 WER 约为：

设置 WER
添加全身关键点 56.1501%
无全身关键点 53.3210%

这个结果说明，并不是加入更多关键点就一定更好。全身关键点可能带来噪声、尺度不一致或冗余维度。如果模型没有显式利用人体拓扑结构，更多点反而可能干扰手部动作的精细建模。

2025.08.08

这一周主要是工程和数据问题：

  • video loader 改进，3000 帧视频读取从约 30s 降到约 6s；
  • 发现中文数据的 keypoint 和 RGB 索引似乎对不上，需要继续处理；
  • Two Stream 训练很慢，大约 10 小时以上一个 epoch。

这些问题看似工程细节，但对研究很关键：长视频、RGB-pose 对齐和训练吞吐量会直接限制实验迭代速度。

阶段六：从识别转向动作 token 与 LM 输入

2025.09.09 - 2025.09.11

后续开始出现 RVQ 训练和动作 token 化方向。记录里提到：

  • block 无重叠；
  • chunk size = 10；
  • 继续检查 dataset 归一化；
  • 可视化 decode 效果。

这说明研究目标开始从“CTC 识别 gloss”扩展到“把连续手语动作编码成离散 token”。这一步很自然：如果想把手语动作接入语言模型，连续关键点必须被压缩成可序列化、可预测、可重建的 token 表示。

RVQ 的价值在于把连续动作表示离散化。一个残差量化器先用第一层 codebook 近似输入，再用后续 codebook 逐层拟合残差。这样可以用多个离散 token 表示一个动作片段，既保留压缩能力，也适合接入自回归模型。

阶段七：长视频与文档级评价

后期还有两个分支。

长视频训练素材

记录中开始收集长视频训练素材，例如央视新闻频道节目。这说明任务可能不再局限于短句级识别，而是考虑更长、更真实的视频输入。长视频会带来几个新问题：

  • 视频读取和缓存效率；
  • 长序列 chunking；
  • 跨 chunk 的动作连续性；
  • 句子/段落边界；
  • 识别或生成结果的篇章级一致性。

Document-level metrics

传统 BLEU 对短句翻译常用，但对长文本会失效。比如每句话内部词都对，但句子顺序全乱，普通 BLEU 可能仍然给出不低分数。记录中关注了两个方向：

  • BlonDe：面向 document-level evaluation，把文本拆成 document、paragraph、sentence、token 等层级，在不同层级上综合 BLEU 和顺序一致性惩罚；
  • TER：Translation Edit Rate，不只考虑 insert/delete/substitute，还考虑 shift，适合评价“信息都在但顺序不同”的情况。

这条线和长视频训练素材是连着的：如果模型未来输出长段文本，只看 WER 或句级 BLEU 不够，需要文档级指标评价跨句顺序、信息组织和篇章结构。

当前阶段性结论

1. CTC 不是唯一瓶颈

CTC 有 peaky prediction 和弱对齐问题，但人为构造良好 feature 后，简单分类器也能产生较好的 decode。这说明当前最大瓶颈很可能在 feature 表示，而不是 CTC 公式本身。

2. 关键点预处理非常敏感

mean/std、中心化方法、cover ratio、不同部位尺度都会改变动作差异。虽然已有中心化对比的 silhouette score 差异不大，但这些实验说明必须把 preprocessing 当成模型的一部分看待。

3. 人体结构先验需要显式建模

简单 MLP + Transformer 很可能没有充分利用骨架拓扑和左右手/身体的结构关系。body part embedding、ST-GCN stream、多部位 VAE 和 Sign Token 分部位排列，都是在往这个方向靠。

4. 识别、对齐、token 化是同一条链

早期的 CTC/force alignment 解决的是 gloss 边界问题；后期的 RVQ/Sign Token 解决的是动作离散表示问题。两者不冲突：前者帮助理解动作与 gloss 的时间关系，后者帮助把动作表示接入 LM。

5. 长视频会要求新的工程和评价体系

短句级 WER 仍然重要，但长视频场景还需要高效 loader、chunk-level tokenization、跨 chunk 建模，以及 BlonDe/TER 这类 document-level metrics。

后续工作建议

  1. 先固定一个强 baseline
    至少保留 Two Stream 或 ST-GCN 级别的对照，避免只在自建 pure CTC 结构里循环调参。

  2. 系统做 feature 可分性诊断
    对每个版本输出 silhouette score、同类/异类 cosine distance、UMAP/t-SNE 可视化和 CTC WER，建立 feature quality 与最终识别性能的对应关系。

  3. 把 preprocessing 纳入 ablation 表
    系统比较中心化方式、尺度归一化、std 缩放、部位加权、cover ratio、是否保留全身关键点，而不是零散可视化。

  4. 优先验证部位结构建模
    继续推进 ST-GCN stream、body/hand 分支、part embedding、多部位 VAE。手语识别里左右手和身体姿态不是等价坐标点，结构先验应该显式进入模型。

  5. 改进 decode 和 alignment，但不要让它背锅
    beam size、language model rescoring、blank penalty、force alignment 的开始/结束边界规则都值得做，但前提是概率矩阵和 feature 已经足够可分。

  6. 将 Sign Token 序列化和 LM 训练目标打通
    当前 body/lhand/rhand 交错排列是合理的，但需要继续验证：这种排列是否比按部位分块、按量化层分块、或按 chunk 分块更利于 LM 学习。

  7. 为长视频建立评价协议
    短句用 WER/CER，长文本补充 BlonDe、TER 或类似 document-level metrics，并记录句子边界、段落顺序和信息遗漏。
